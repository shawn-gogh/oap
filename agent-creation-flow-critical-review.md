# 智能体创建流程批判性评审

## 1. 总体判断

当前智能体创建流程功能丰富，但核心问题不是能力数量不足，而是界面所承诺的智能程度、安全性和可验证性，高于系统真正执行的程度。

目前的产品更接近一个高级智能体配置生成器，还不是一个能够证明智能体已连接真实资源、受到可靠治理并可稳定完成实际工作的上线向导。

本评审基于当前版本的前端流程、状态管理、创建接口及运行时配置逻辑。由于评审时没有可连接的内置浏览器，视觉呈现和真实点击体验仍建议补充一次用户走查。

## 2. 高优先级问题

### 2.1 “自动推荐模型”可能不会采用推荐结果

创建首页告诉用户，系统会根据任务自动推荐模型。但生成配置后，如果页面原先已有选中模型，代码会强制将生成结果中的模型替换为原模型。

代码位置：`src/ui/src/app/agents/new/page.tsx`

```ts
const nextDraft = selectedModel
  ? { ...generatedDraft.draft, model: selectedModel }
  : generatedDraft.draft;
```

这意味着即使生成模型判断另一个模型更合适，推荐结果也会被静默覆盖。用户看到的是“自动推荐”，实际得到的是“沿用默认值”。

建议：

- 区分当前默认模型与 AI 推荐模型。
- 展示推荐理由、成本、上下文长度和工具支持差异。
- 由用户明确确认是否采用推荐模型。

### 2.2 “写操作需审批”不是不可绕过的强制审批

复核页告诉用户，写操作或破坏性操作会暂停并等待人工确认。但当前实现主要是在智能体配置中挂载 `request_human_approval` MCP。

代码位置：`src/ui/src/lib/agent-builder.ts`

这种实现仍依赖模型主动选择审批工具。`bash`、`write`、`edit` 等工具本身没有被独立策略层强制拦截。因此它更接近提示模型审批，而不是平台保证审批。

这是安全和用户信任层面的高风险问题。

建议：

- 在工具执行边界判断操作的副作用级别。
- 对写入、删除、外部发送等操作强制校验 approval token。
- 没有有效审批时，运行时直接拒绝工具执行。
- 在界面上区分提示性治理与平台强制治理。

### 2.3 创建前试跑无法验证真实智能体能力

创建前试跑只测试 system prompt 与模型，不挂载工具、MCP 和凭证。

代码位置：

- `src/ui/src/app/agents/new/review-step.tsx`
- `src/ui/src/lib/api.ts`

这对研究、邮件、数据分析、事故处理等智能体尤其失真，因为这些智能体的核心价值来自外部数据和工具。纯模型对话通过，并不能证明真实任务可以完成。

同时，不论用户选择 `rule`、`llm_judge` 还是 `environment`，创建前试跑实际都使用模型进行判断，容易产生以下问题：

- 被测模型和评估模型相同，偏差高度相关。
- `rule` 没有执行确定性规则。
- `environment` 没有验证真实环境。
- 测试通过后仍可能无法读取数据、调用工具或完成写操作。

建议提供三层测试：

1. 静态配置与依赖检查。
2. 真实只读工具冒烟测试。
3. 隔离环境中的端到端任务测试。

### 2.4 系统允许创建注定运行失败的智能体

前端创建校验目前主要检查：

- 名称是否为空。
- 是否选择模型。
- 是否选择 Runtime。
- System prompt 是否为空。
- Vault key 名称格式是否合法。

代码位置：`src/ui/src/app/agents/new/builder-shared.ts`

当前没有阻止以下情况：

- Runtime 未连接或已经离线。
- 模型在所选 Runtime 中不可用。
- MCP 缺少凭证。
- Vault 只声明变量名，但没有实际密钥。
- 工具不受当前 Runtime 支持。
- 子智能体不存在、不可访问或形成循环依赖。
- 定时任务依赖的资源尚未配置。

后端创建校验更弱，主要只检查名称、owner 和非空 Runtime。

代码位置：`src/db/managed_agents/registry/input.rs`

这会把配置错误推迟到运行阶段，最终表现为“创建成功但智能体不好用”。

建议引入明确的生命周期：

- `Draft`：允许保存不完整配置。
- `Ready`：所有关键依赖预检通过。
- `Active`：允许人工运行或调度。
- `Degraded`：依赖失效，暂停或限制运行。

“创建成功”不应等于“已经可运行”。

## 3. 用户体验问题

### 3.1 Fit 步骤没有真正判断是否应该创建智能体

第一步叫“定位 Fit”，但主要交互是自由文本、模板选择和生成按钮。它没有系统地帮助用户明确：

- 输入来自哪里。
- 输出交付给谁。
- 多久运行一次。
- 是否需要持续状态。
- 哪些操作可以自动执行。
- 错误如何发现和恢复。
- 为什么应该使用智能体，而不是提示词或固定工作流。

更合理的 Fit 阶段应给出明确结论：

- 适合智能体。
- 更适合固定工作流。
- 更适合单次模型调用。
- 当前条件不足，暂不建议创建。

同时说明判断理由和需要补充的条件。

### 3.2 评估步骤过早，容易退化成形式主义

当前顺序是：

```text
Fit → Eval → Design → Review
```

用户还没有看清模型、工具、MCP、技能和系统提示词，就被要求定义成功标准和四类测试用例。普通用户很难评估一个尚未理解的设计。

系统为了降低阻力，又提供“填入建议用例”。这使评估门禁很容易退化为点击默认模板即可通过的流程税。

建议调整为：

```text
目标与边界 → 初步设计 → 关键评估标准 → 真实试跑 → 复核上线
```

### 3.3 可行性四问默认全选，选择“否”也不改变行为

默认配置将复杂度、价值、模型适配度和错误可恢复性全部设为 `true`。用户取消某些选项后，系统只显示提示，仍然可以继续。

它不会自动执行以下调整：

- 简化为单次模型调用。
- 移除自主循环。
- 缩减工具权限。
- 降低最大运行时间。
- 推荐固定工作流。

因此可行性检查看起来具有方法论，但对最终系统行为影响很小。

建议让可行性结果直接影响设计方案，并要求用户确认偏离建议的原因。

### 3.4 对普通用户暴露过多底层概念

一个创建流程同时出现：

- Runtime
- Harness
- Model
- Tools
- Skills
- Rules
- MCP integrations
- Vault keys
- Sub-agents
- YAML/config
- Governance
- Evaluator

这些概念的边界并不天然清晰，特别是：

- Tool 与 MCP tool 的区别。
- Skill 与 Rule 的区别。
- Runtime 与 Harness 的区别。
- Vault key 声明与实际凭证的区别。
- System prompt 与生成配置的关系。

建议默认采用任务导向模式，只让用户配置：

- 目标。
- 数据来源。
- 允许执行的动作。
- 运行和触发方式。
- 结果交付位置。
- 人工确认边界。

底层 Runtime、MCP、工具和 YAML 放入高级模式。

### 3.5 对话、UI 和 YAML 三套编辑方式缺少明确权威关系

当前系统支持对话修改、表单修改和 YAML 修改，功能灵活，但用户不容易判断：

- 哪一种配置是最终权威来源。
- 对话修改是否会覆盖手工字段。
- YAML 修改后 UI 是否完整保留。
- 切换 Runtime 是否会重置工具和模型。
- 模型生成是否会修改用户未要求修改的字段。

建议每次 AI 修改都提供字段级 diff，允许用户逐项接受或拒绝，而不是直接改写整个配置。

## 4. 智能体实用性问题

### 4.1 默认权限过宽，缺少任务驱动的最小授权

空白智能体默认包含：

- `bash`
- `read`
- `write`
- `edit`
- `glob`
- `grep`
- `web_fetch`
- `web_search`

代码位置：`src/ui/src/lib/agent-builder.ts`

这既扩大安全风险，也会降低智能体的选择稳定性。工具越多，模型越容易选错工具，同时增加延迟与成本。

建议默认从零权限或最小只读权限开始，根据任务推导最小工具集合，并对每项权限解释：

- 为什么需要。
- 可以访问什么。
- 可能产生什么副作用。
- 是否需要人工审批。

### 4.2 模板描述了能力，但没有建立真正的数据连接

邮箱、支持、事故处理和数据分析模板会预填 `GMAIL_API_KEY`、`DATABASE_URL` 等名称，但这只是凭证变量声明，不代表：

- 对应 MCP 已挂载。
- 所需工具真实存在。
- 凭证已经保存。
- 权限范围正确。
- API 连接测试通过。

这容易制造“选择模板后即可工作”的错觉。

建议将模板升级为可安装方案，至少包含：

- 所需数据源。
- MCP 或工具依赖。
- 凭证设置向导。
- 最小权限建议。
- 连接测试。
- 示例任务和预期结果。

### 4.3 缺少输入、输出和交付渠道设计

当前流程很重视模型如何工作，但不够关注真实业务闭环：

- 谁触发智能体。
- 输入来自聊天、定时器、Webhook、邮箱还是 Slack。
- 结果保存或发送到哪里。
- 谁负责确认结果。
- 失败后通知谁。
- 下次运行是否继承状态。
- 如何避免重复执行。

一个实用智能体的关键通常不是 system prompt，而是触发、上下文、权限、交付与恢复机制。

### 4.4 缺少成本、时延和资源预算

用户选择模型、运行时、最大运行时间和子智能体时，看不到：

- 单次预估成本。
- 最大可能成本。
- 典型执行时间。
- 子智能体调用的成本放大倍数。
- 定时任务每月运行次数。
- 模型和工具限额。

因此用户无法判断智能体是否值得长期运行，Fit 阶段的价值判断也没有形成闭环。

### 4.5 创建成功后缺少上线验收

当前创建成功后直接进入详情页。更完整的上线验收应包括：

- 配置完整度。
- Runtime 连通性。
- 模型可用性。
- 凭证状态。
- 工具冒烟结果。
- 首次真实任务结果。
- 调度是否启用。
- 失败通知是否配置。

用户真正关心的不是对象是否已经写入数据库，而是智能体现在能否完成工作。

## 5. 一致性和可维护性问题

### 5.1 页面显示的创建接口与实际接口不一致

流程顶部显示 `POST /v1/agents`，但实际创建接口是 `/api/agents`。

代码位置：`src/ui/src/app/agents/new/steps-bar.tsx`

这会误导技术用户，也增加文档和排障成本。

### 5.2 中英文和术语不统一

页面同时使用 `Fit`、`Eval`、`Design`、`Review`、`Runtime`、`Vault Credentials`、`Tools`、`Skills` 等英文术语和中文说明。

建议建立统一术语表，至少保证同一流程内的语言和概念命名一致。

### 5.3 资源加载失败可能被误认为资源为空

部分 Runtime、Agent、Skill 和 Rule 请求失败时会降级为空数组。用户可能把接口失败理解为系统没有可用资源。

建议明确区分：

- 数据为空。
- 数据仍在加载。
- 接口请求失败。
- 当前用户没有权限。

## 6. 改进优先级

### P0：消除虚假确定性和安全风险

1. 将人工审批改为工具执行层的强制策略。
2. 创建前验证 Runtime、模型、凭证、MCP 和工具兼容性。
3. 让试跑覆盖真实工具链，而不是仅测试模型回答。
4. 修复自动推荐模型被默认模型覆盖的问题。
5. 不允许缺少关键依赖的智能体直接进入 Active 状态。

### P1：提高创建流程效率和可理解性

1. 将 Fit 改为是否需要智能体的真实诊断。
2. 先展示初步设计，再定义评估标准。
3. 默认隐藏 Runtime、Harness 和 YAML 等底层细节。
4. 使用字段级 diff 管理 AI 配置修改。
5. 将创建过程拆分为 Draft、Ready 和 Active 状态。

### P2：提高长期业务价值

1. 将模板升级为包含数据源、凭证、工具、触发器和交付渠道的完整方案。
2. 增加成本、时延和资源预算。
3. 增加首次真实任务和上线验收清单。
4. 补充运行失败通知、恢复策略和重复执行控制。
5. 建立真实任务反馈到评估集的持续改进闭环。

## 7. 建议的目标流程

```text
描述业务目标
  ↓
Fit 诊断：Agent / Workflow / 单次调用
  ↓
识别输入、输出、数据源、触发器和副作用
  ↓
生成最小权限的初步设计
  ↓
连接 Runtime、模型、MCP 和凭证
  ↓
定义关键成功标准
  ↓
静态预检 + 工具冒烟 + 隔离环境试跑
  ↓
治理与成本复核
  ↓
保存为 Draft / 激活为 Active
  ↓
首次任务验收与持续评估
```

## 8. 结论

当前系统很擅长生成一份看起来完整的智能体配置，但还不够擅长证明这个智能体已经连接真实资源、受到可靠治理，并能够稳定完成用户的实际工作。

下一阶段产品改进的重点不应只是继续增加更多配置项，而应缩小以下差距：

```text
配置已生成 ≠ 依赖已就绪 ≠ 行为受治理 ≠ 真实任务可完成
```

只有当创建流程能够验证这四个阶段，用户才会把“创建成功”理解为“这个智能体真的可以投入使用”。
